# 中文模型接入流程说明

## 1. 说明目的

本文档用于说明当前项目如何将中文 GPT-2 模型转换为浏览器可加载的前端资产，并接入现有的 Transformer 可视化系统。

该流程面向当前仓库设计，遵循以下原则：

1. 不重构现有前端可视化结构。
2. 通过离线脚本生成模型资产。
3. 通过统一配置文件保证 Python 脚本与前端参数一致。
4. 保证后续更换中文模型时可以重复执行同一套流程。

## 2. 流程总览

整个中文模型接入流程分为四个阶段：

1. 下载中文预训练模型。
2. 导出前端兼容的 ONNX 模型。
3. 将 ONNX 模型切分为浏览器分片。
4. 生成前端使用的中文示例数据。

整体关系如下：

```text
01.py
  -> 下载中文 HuggingFace 模型

02_convert_onnx.py
  -> 将 PyTorch 权重映射到项目自定义 GPT 架构
  -> 导出前端兼容的 model.onnx

03_chunk.py
  -> 将 model.onnx 切分为 gpt2.onnx.partN
  -> 输出到 static/model-v2-chinese

04_gen_examples.py
  -> 基于 ONNX 模型生成 ex0 ~ ex4 中文示例数据

前端
  -> 读取统一配置
  -> 加载 tokenizer
  -> 合并模型分片
  -> 创建 ONNX Runtime session
  -> 展示推理与可视化结果
```

## 3. 统一配置文件

当前流程的统一配置文件为 [model-config.json](model-config.json)。

它负责集中管理以下内容：

1. 模型来源配置
2. tokenizer 配置
3. 本地模型目录
4. ONNX 输出目录
5. 浏览器静态模型目录
6. 前端访问模型目录
7. 分片前缀、分片大小、分片总数
8. 缓存版本
9. 默认温度、top-k、示例数据和随机种子

它的作用是避免前端和 Python 脚本分别维护两套模型参数。

## 4. 各阶段详细说明

### 4.1 Step 1: 下载中文模型

执行脚本：[01.py](01.py)

主要功能：

1. 读取 [model-config.json](model-config.json) 中的模型 ID、本地保存目录和代理配置。
2. 调用 `snapshot_download` 从 HuggingFace 下载中文 GPT-2 模型。

当前默认模型：

- `uer/gpt2-chinese-cluecorpussmall`

默认输出目录：

- `models/gpt2-chinese-cluecorpussmall`

该阶段完成后，应至少确认以下文件存在：

1. `config.json`
2. 模型权重文件
3. `vocab.txt`
4. tokenizer 相关文件

这一阶段的产物是后续 ONNX 导出的输入。

### 4.2 Step 2: 导出 ONNX 模型

执行脚本：[02_convert_onnx.py](02_convert_onnx.py)

主要功能：

1. 读取 [model-config.json](model-config.json)。
2. 读取本地模型目录中的 `config.json`。
3. 校验层数、头数、隐藏维度、词表大小是否与共享配置一致。
4. 复用 [src/utils/model/model.py](src/utils/model/model.py) 中的自定义 GPT 架构。
5. 将 HuggingFace 权重手动映射到该架构中。
6. 导出为前端可兼容的 `model.onnx`。
7. 使用 onnxruntime 做一次推理验证。

默认输出目录：

- `models/gpt2-chinese-onnx/model.onnx`

该步骤的重点不是只生成一个 ONNX 文件，而是保证导出的输出张量名称与当前前端逻辑完全一致。也就是说，前端现有 attention 可视化、logits 展示和 softmax 采样逻辑不需要因为中文模型而重写。

### 4.3 Step 3: 切分模型分片

执行脚本：[03_chunk.py](03_chunk.py)

主要功能：

1. 读取共享配置中的 ONNX 路径、静态资源目录、chunk 大小和分片前缀。
2. 将 `model.onnx` 按固定大小切分为多个小文件。
3. 输出到浏览器静态目录。
4. 校验实际分片数是否与共享配置中的 `chunkTotal` 一致。

默认输出目录：

- `static/model-v2-chinese`

默认命名格式：

- `gpt2.onnx.part0`
- `gpt2.onnx.part1`
- `gpt2.onnx.part2`
- `...`

这一阶段的产物会被前端页面直接按 URL 拉取。

### 4.4 Step 4: 生成中文示例数据

执行脚本：[04_gen_examples.py](04_gen_examples.py)

主要功能：

1. 读取共享配置中的中文示例 prompt。
2. 使用 ONNX Runtime 对每条 prompt 执行推理。
3. 提取前端可视化所需数据。
4. 生成 ex0 到 ex4 的中文示例文件。

生成字段包括：

1. `prompt`
2. `tokens`
3. `tokenIds`
4. `logits`
5. `outputs`
6. `probabilities`
7. `sampled`

默认输出目录：

- `src/constants/examples`

当前脚本已固定随机种子，因此生成结果在同一模型和同一配置下具有可重复性。

## 5. 前端接入方式

### 5.1 运行时配置读取

[src/store/index.ts](src/store/index.ts) 会读取 [model-config.json](model-config.json)，并生成当前模型的运行时配置，包括：

1. `tokenizerId`
2. `modelDir`
3. `chunkFilePrefix`
4. `chunkTotal`
5. `cacheVersion`
6. 默认中文示例文本

### 5.2 模型加载入口

[src/routes/+page.svelte](src/routes/+page.svelte) 会在页面启动时完成：

1. 加载 tokenizer
2. 根据运行时配置拼接模型分片 URL
3. 调用分片加载逻辑合并模型
4. 创建 ONNX Runtime session

### 5.3 分片缓存机制

[src/utils/fetchChunks.js](src/utils/fetchChunks.js) 使用缓存版本号管理模型缓存。

这意味着如果模型重新导出或重新分片，就应该同步调整 `cacheVersion`，否则浏览器可能继续使用旧分片。

### 5.4 推理与可视化消费

[src/utils/data.ts](src/utils/data.ts) 负责：

1. 对输入文本进行编码
2. 执行 ONNX 推理
3. 读取 attention 输出和 `linear_output`
4. 计算 temperature、top-k、top-p 对应的概率分布
5. 将结果交给现有前端组件展示

## 6. 推荐执行顺序

建议严格按如下顺序操作：

```bash
python 01.py
python 02_convert_onnx.py
python 03_chunk.py
python 04_gen_examples.py
```

完成后再启动前端：

```bash
npm run dev
```

## 7. 验收要点

流程执行完成后，建议按以下清单验证：

1. 模型目录是否下载完整。
2. `model.onnx` 是否成功导出。
3. ONNX 是否能通过 onnxruntime 验证推理。
4. 分片文件是否全部生成。
5. 分片数量是否与共享配置一致。
6. ex0 到 ex4 是否已更新为中文示例。
7. 页面是否能正常加载 tokenizer。
8. 页面是否能正常加载模型分片。
9. 中文输入是否能正常分词和预测。
10. 注意力、logits、概率分布是否正常显示。
11. 刷新页面后是否存在旧缓存污染。

## 8. 主要风险

### 8.1 tokenizer 与词表不匹配

如果 tokenizer 与模型输出 vocab 大小不一致，会导致 token 解码错误、概率异常或采样失真。

### 8.2 分片数量不一致

如果 `03_chunk.py` 的实际输出数量与共享配置中的 `chunkTotal` 不一致，前端会在合并模型时失败。

### 8.3 浏览器缓存使用旧分片

如果更新了模型文件却没有同步更新缓存版本，页面可能继续使用旧模型。

### 8.4 示例数据未同步刷新

如果重新导出了 ONNX，但没有重新生成 ex0 到 ex4，页面首屏示例会和真实模型行为不一致。

## 9. 方案特点

该方案的优势在于：

1. 不动现有前端主要可视化结构。
2. 模型接入逻辑集中在统一配置与离线脚本。
3. 更换模型时不需要到处手工改常量。
4. 适合后续重复执行、维护和交接。

适用场景包括：

1. 将英文 GPT-2 替换为中文 GPT-2。
2. 在浏览器中部署中文 Transformer 可视化模型。
3. 需要一套可复现的模型资产构建流程。
4. 需要保证教学示例与真实模型输出一致。
